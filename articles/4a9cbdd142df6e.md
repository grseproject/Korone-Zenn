# #52 ガバナンス文書でオーケストレーターを“法治”する仕組み（実装編）

## 🍔 はじめに
Claude Codeを使ったマルチエージェント開発も、エージェント数が10体を超えたあたりから全く新しいフェーズに突入しました。
初期は「いかに個別のエージェントに正しく仕事をさせるか」が最大の課題でしたが、システム全体が成熟してくると、今度は**「全体を束ねるオーケストレーター（Claude自身）がルールを破り始める」**という新たな問題に直面するようになりました。
本記事では、増えすぎたルールによるオーケストレーターの「認知負荷」を下げ、自律的に自己矯正（Behavioral Constraints）を行わせるための**「ガバナンス文書（Governance Docs）」**の導入について解説します！
## 🧭 前提：暗黙知とプロンプトの限界
2月までの安定化フェーズでは、ルールや制約は主に以下のような場所で管理されていました。
 1. 各エージェントの SKILL.md 内の注意事項
 2. 実行時のプロンプト（「〜〜しないでください」「〜〜の形式で出力してください」）
 3. 汎用的なシステムプロンプト
これらはエージェントが少ないうちはうまく機能していました。しかし、エージェントが13体、サブスキルが13以上になり、連携が複雑化すると限界を迎えます。
## 💀 問題：オーケストレーターの「指示無視」と「勝手な判断」
ルールが20個を超えたあたりから、ディスパッチャーとして振る舞うClaude Codeに以下のようなインシデントが見られ始めました。
:::message alert
**実際に起きたインシデント**
 * **パスの捏造**: 存在しないファイルを「ある」と仮定して次のエージェントに渡してしまう。
 * **ルールの忘却**: 「必ずJSONパスのみを渡す」というルールを忘れ、勝手に中身をMarkdownで要約して渡す（※以前の記事で非介入Hookを作った原因です）。
 * **役割の越権**: 本来はQuality Reviewerに回すべき承認プロセスを、オーケストレーター自身が「ヨシ！」とパスしてしまう。
   :::
**要するに、ルールが多すぎてClaudeのコンテキストウィンドウやアテンションがパンクし、勝手に作業をショートカットし始めたのです。**
## 🏛️ 解決策：「ガバナンス文書」による法治国家化
この問題に対処するため、都度プロンプトで指示する「人治」から、明文化された憲法に従わせる**「法治」**へとアーキテクチャをシフトしました。
具体的には、運用ルールを「ガバナンス文書」として外部化し、オーケストレーターの思考プロセスに組み込みました。

### 1. 4つのガバナンス文書の制定
Quality Gates Rule

各作業エージェントは、最終成果物を生成する前に必ず Draft（設計方針・概要） を作成し、
それを対象の Reviewer エージェントに提出しなければならない。

Reviewer からの Approve を受け取るまでは、本実装の生成フェーズに移行してはならない。


### 2. Reviewer エージェントのプロンプト改修
Code Reviewer や Quality Reviewer の `SKILL.md` にも手を加え、完成品だけでなく「計画書（Draft）」をレビューするモードを追加しました。

:::details Reviewerのプロンプト追加部分
```markdown
# Role
あなたはシステムの品質を担保する Reviewer です。

## Pre-Review Mode (事前審査)
もし入力された内容が「設計」や「計画」のドラフトである場合、以下の観点で審査してください。
- アーキテクチャ規約（依存関係など）に違反する計画になっていないか？
- エッジケースの考慮漏れはないか？

問題があれば、実装に入る前に方針の軌道修正を指示してください。


:::

🎯 効果：トークン消費の削減と「手戻りゼロ」へ

この Pre-Review Model を導入した結果、3月の統計では驚くべき変化がありました。

コードを書き始める前に方針を正すため、「書いては消し」の無駄なトークン消費が激減しました。また、作業エージェント自身も「承認された計画」に沿って作業を進めるため、出力のブレが圧倒的に少なくなりました。

📝 まとめ

AIエージェントに作業を任せるとき、人間と同じように**「方向性のすり合わせ（事前レビュー）」**が不可欠です。

❌ 成果物が完成してからレビューする（トークンと時間の無駄）

✅ 作業着手前にドラフトを事前審査（Pre-Review）する

これにより、システム全体の品質保証が「リアクティブ（事後）」から「プロアクティブ（事前）」へと進化し、より洗練された自律ネットワークへと成長しました。

さて、システムが高度化する一方で、3月にはもう一つの面白い変化がありました。それは**「全エージェントにキャラクター名（固有名）を付ける」**という工夫です。

次回は、この「名付け」がエージェントネットワークにどのような（意外な）好影響を与えたのかについて書きたいと思います！🚀
