# #41 Test Scope Generator の内部構造（実装編）  
**― “入口エージェント”が完成するまでの試行錯誤と、最終構造 ―** 📝

Test Scope Generator は、  
ネットワーク全体の“入口”として最も重要なエージェントだった。

しかし最初から今の形だったわけではない。  
むしろ初期は **不安定・揺れる・差し戻される・形式が揃わない** という問題だらけだった。

このエージェントが安定するまでに、  
- 3回の構造刷新  
- 2回の責務分割  
- 1回の完全書き直し  
を経ている。

この記事では、  
**その進化の背景と、最終的に落ち着いた内部構造** をまとめる。

---

# 🧩 1. 初期の問題：生成物が揺れすぎた

初期の Test Scope Generator は、  
「とりあえずスコープを作る」だけの存在だった。

```
extract → build → output
```

しかしこの構造には致命的な問題があった。

### ■ 問題1：要件の粒度が揃わない  
チケットの書き方がバラバラなので、  
生成されるステップもバラバラ。

### ■ 問題2：Reviewer が読みづらい  
Markdown の形式が揺れ、  
Quality Reviewer のスコアが安定しない。

### ■ 問題3：TestRail Integrator が ingest できない  
JSON の形式が毎回微妙に違う。

### ■ 問題4：例外系が抜ける  
acceptance criteria に書かれていない例外系は生成されない。

---

# 🧭 2. 進化の方向性が決まった瞬間

問題が積み上がった結果、  
Test Scope Generator は **“入口の品質が全体の品質を決める”**  
という事実が明確になった。

そこで方向性が決まる。

> **「生成」ではなく「整える」エージェントにする。**

つまり、  
- 要件を整える  
- ステップを整える  
- 出力を整える  
- Reviewer が読みやすい形に整える  
- Integrator が ingest しやすい形に整える  

**“整える” が中心になった。**

---

# 🧱 3. 最終構造（6モジュール構成）

```
extract_requirements
normalize_requirements
build_steps
enrich_steps
output_md
output_json
```

この6つが揃って初めて、  
Test Scope Generator は安定した。

---

# 🧠 4. モジュールごとの背景と実装

## ① extract_requirements  
**背景：チケットの書き方が揺れるので、まず標準化が必要だった。**

```python
def extract_requirements(ticket):
    return {
        "title": ticket["title"],
        "description": ticket.get("description", ""),
        "acceptance": ticket.get("acceptance_criteria", []),
        "priority": detect_priority(ticket),
        "tags": ticket.get("labels", [])
    }
```

---

## ② normalize_requirements  
**背景：Reviewer のスコアが揺れる原因が“粒度の揺れ”だった。**

```python
def normalize_requirements(reqs):
    normalized = []
    for ac in reqs["acceptance"]:
        ac = ac.strip().rstrip(".")
        normalized.append(ac)
    reqs["acceptance"] = normalized
    return reqs
```

---

## ③ build_steps  
**背景：最低限のステップを確実に生成するための“基礎”。**

```python
def build_steps(reqs):
    return [
        {"action": f"Verify: {ac}", "expected": f"{ac} is satisfied"}
        for ac in reqs["acceptance"]
    ]
```

---

## ④ enrich_steps  
**背景：例外系と前提条件の抜け漏れが多かった。**

```python
def enrich_steps(steps, reqs):
    enriched = [{
        "action": "Ensure system is in initial state",
        "expected": "Initial state confirmed"
    }]
    enriched.extend(steps)
    if "error" in reqs["tags"]:
        enriched.append({
            "action": "Trigger error condition",
            "expected": "Error is handled gracefully"
        })
    return enriched
```

---

## ⑤ output_md  
**背景：Reviewer が読みやすい形式に統一する必要があった。**

```python
def output_md(reqs, steps):
    md = f"# {reqs['title']}\n\n## Steps\n"
    for i, s in enumerate(steps, 1):
        md += f"{i}. **Action:** {s['action']}\n"
        md += f"   - Expected: {s['expected']}\n"
    return md
```

---

## ⑥ output_json  
**背景：TestRail Integrator が ingest できる形式に統一。**

```python
def output_json(reqs, steps):
    return {
        "title": reqs["title"],
        "priority": reqs["priority"],
        "steps": steps
    }
```
