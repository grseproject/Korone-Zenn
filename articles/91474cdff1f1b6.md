# #34 判断の“重心”はどこにあったのか？（実装編）  
**― エージェントネットワークの中で、どこが判断を担っていたのか ―** 🎯

エージェントが単体ツールからネットワークへと進化していくと、  
自然と **「判断の中心」** が生まれる。

今回は、実際のコード断片を交えながら、  
**どの役割が判断の重心になっていたのか** を整理していく。

---

## 🎯 1. 初期の重心は「生成」

最初に中心にいたのは **Test Scope Generator**。  
すべての流れがここから始まるため、  
自然と判断が集中していた。

実際の生成ロジックはシンプルで、  
「要件を読み取ってスコープを組み立てる」ことに集中していた。

```python
def generate_scope(ticket):
    reqs = extract_requirements(ticket)
    scope = build_scope(reqs)
    return {
        "scope": scope,
        "priority": detect_priority(reqs)
    }
```

この段階では、  
**生成＝判断** という構造だった。

---

## 🔍 2. 次に重心が移ったのは「品質」

生成が安定してくると、  
判断の中心は **Quality Reviewer** と **Code Reviewer** に移動した。

特に Code Reviewer の PreToolUse Hook は、  
ネットワーク全体の“安全装置”として機能した。

```python
def pretooluse_hook(operation):
    issues = analyze_code_change(operation.diff)
    score = score_change(issues)

    if score < 90:
        return {"allowed": False, "score": score, "issues": issues}

    return {"allowed": True}
```

ここで初めて、  
**「作る」より「正しく作る」ことが中心になる**。

---

## 📚 3. 仕様が整うと「仕様」が重心になった

Spec Manager が強化されると、  
判断の中心は **仕様の正しさ** に移った。

Wiki との差分検出はこう動いていた。

```python
def detect_spec_diff(local, wiki):
    diff = []
    for key in local:
        if local[key] != wiki.get(key):
            diff.append({"field": key, "local": local[key], "wiki": wiki.get(key)})
    return diff
```

差分があれば、  
自動更新かレビュー依頼かを判断する。

```python
def handle_diff(diff):
    if not diff:
        return "no_update"
    if is_minor(diff):
        return "auto_update"
    return "needs_review"
```

この時期は、  
**仕様がネットワークの“基準点”** になっていた。

---

## 🧠 4. ネットワークが成熟すると「分析」が重心に

Data Analyzer が多スキル化すると、  
判断の中心は **状況を読むこと** に移動した。

Issue Triage の判断ロジックはこう。

```python
def triage(issue):
    signals = collect_signals(issue)
    severity = estimate_severity(signals)
    impact = estimate_impact(signals)
    return {
        "severity": severity,
        "impact": impact,
        "recommendation": recommend_action(severity, impact)
    }
```

ここでは、  
**判断のほとんどが分析側に寄っていた**。

---

## 🗂️ 5. 最後に重心が落ち着いたのは「安定性」

ネットワークが完成に近づくと、  
中心にいたのは **Session Manager**。

バックアップと復旧はこう動いていた。

```python
def backup_session(session):
    path = build_backup_path(session.id)
    save_json(path, session.data)
    return path
```

```python
def restore_session(session_id):
    path = find_latest_backup(session_id)
    if not path:
        return None
    return load_json(path)
```

最終的には、  
**「判断」より「継続性」** が中心になった。

---

## 🌈 重心の移動をまとめると

| フェーズ | 重心 | 理由 |
|:---|:---|:---|
| 初期 | 生成 | すべての起点 |
| 中期 | 品質 | 正しく作ることが中心に |
| 中盤 | 仕様 | 真実の基準が必要に |
| 後半 | 分析 | 状況を読む力が中心に |
| 完成 | 安定 | ネットワークの生命線 |

コードを見ると、  
**どこに判断が集まっていたかが自然に見えてくる。**
