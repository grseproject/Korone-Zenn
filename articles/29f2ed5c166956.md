## はじめに

Agent開発をしていると、こんな違和感に出会うことがある。

**「機能を足せば足すほど、Agent の輪郭が曖昧になっていく」**

新しい課題が来るたびに、観点を追加し、knowledge を増やし、Agent そのものを増やす。`+`（追加）の方向に思考が偏ると、ある日「このAgentは結局何をするのか」が説明できなくなる。

この記事は、Agent開発の思考を **「数式の演算」というメタファー** で整理したフレームの紹介だ。

`A + B = C` という小学生でも知っている式から始めて、`-` / `×` / `÷` / `^` / `√` まで広げると、Agent設計の判断が驚くほどクリアになる。

数式といっても厳密な代数法則を持ち出すわけではない。**思考の補助線として演算を借りる**、それだけだ。

---

## 基本式

```
A + B = C
```

- **A**: 入力要素（仕様・要件・データ）
- **B**: 処理要素（観点・ルール・手順）
- **C**: 出力要素（成果物・結果）

この3要素を、これから6つの層で組み替えていく。

---

## 第1層: 関係の組み替え（`+` `-`）

同じ3要素でも、視点を変えると3つの式が導かれる。

| 式 | 操作 | 設計上の問い |
|----|------|------------|
| `A + B = C` | 合成 | 何を組み合わせれば価値が生まれるか |
| `C - B = A` | 逆算 | 結果から処理を引けば、本来の入力は何だったか |
| `C - A = B` | 抽出 | 結果から入力を引けば、隠れた処理は何だったか |

ここで重要なのは、**`+` が常に正解とは限らない** という認識だ。

新しい課題があると、つい「Agentを足す」「機能を追加する」と合成方向に走る。しかし本質的な解決は `-`（既存から不要を引いて本質を残す）であることが多い。

数学的にも示唆的だ:

```
+ : 可換、足すのは簡単
- : 非可換、引くのは「何を引くべきか」を見抜く判断力が要る
```

つまり `-` の方が高次の思考を要求する操作なのだ。Agent開発の文脈では、こう言い換えてもいい:

- `+`（追加）= 実装
- `-`（削除・分離）= 設計判断

---

## 第2層: 右辺で「状態」を見る

`A + B - C = 0` のように右辺を眺めると、Agentの状態が分類できる。

| 式 | 状態 | 意味 |
|----|------|------|
| `A + B - C = 0` | 均衡 | 投入と産出が釣り合う（期待通り） |
| `A + B - C = C` | 増幅 | 投入を超える価値が生まれる（理想状態） |
| `A + B - C = -C` | 損失 | コスト割れ（要改善） |
| `A + B - C = B` | 副産物優位 | 成果物は失敗だが処理側の学びが残る |
| `A + B - C = A` | 仕様回帰 | 作って初めて入力の不備が見える |

普通は「Cがどれだけ良いか」だけで評価するが、右辺の形で見ると **「何が残り、何が消えるか」** という別の評価軸が立ち上がる。

「Agentに任せたら成果物の質はイマイチだったけど、観点の整理が進んだ」という体験は、`= B` 状態として捉えられる。失敗ではなく、副産物優位な状態だったのだ。

---

## 第3層: 時間変化（`→`）

要素は時間とともに変化する。変化には2種類ある。

| 変化型 | 表記 | 性質 |
|--------|------|------|
| 改良 | `C → C'` | 同じ役割の中での洗練（連続的） |
| 置換 | `C → D` | 別物への遷移（不連続・世代交代） |

これは A, B, C のいずれにも起こりうる。

```
A → A'  (入力の精緻化)   A → E  (入力源そのものの変更)
B → B'  (処理の改善)     B → F  (処理方式の置換)
C → C'  (出力の改善)     C → D  (出力形式の刷新)
```

ここで運用上重要なのは、**`C'` と `D` の区別を改修前に行う** ことだ。

- `C'` 系の改修: 同じ性質の中での移動 → 版を残せる（バージョンスナップショット有効）
- `D` 系の改修: 性質そのものの遷移 → 版を並べても比較にならない

「ちょっと直すだけのつもり」が、実は性質変化（`→D`）を引き起こしていた。これがAgent運用の典型的な事故パターンだ。

---

## 第4層: 柔軟な定義（型と値の分離）

要素は変化するため、定義は最初から「変化を前提とした柔軟性」を持つ必要がある。

要素の定義は2段階で持つ。

| 段階 | 役割 | 変動性 |
|------|------|--------|
| 型（役割） | 「Aは入力情報である」 | 不変 |
| 値（実体） | 「具体的にはこのフォーマットのデータ」 | 可変 |

型レベルで `A + B = C` が成立していれば、値が変動しても式は破綻しない。

例えば A を「特定の仕様書フォーマット」のような具体物で定義すると、フォーマットが変わった瞬間に式が壊れる。A を「正規化された要件」のように役割で定義すれば、中身が何であっても式は生き残る。

これは設計原理で言うと「**インターフェース安定・実装可変**」と同じ思想だ。

---

## 第5層: 結合と分解（`×` `÷`）

`+` `-` だけでは表せない関係を、`×` と `÷` が補う。

### `×` は三つの意味を持つ

| 演算 | 意味 | 例 |
|------|------|-----|
| `A × n` | 倍率（並列実行） | 同一処理のn並列 |
| `A × B` | 合成（パイプライン） | 生成系 → 検証系 |
| `A × B` | 交差（直積） | 全観点 × 全シナリオ |

### `÷` も三つの意味を持つ

| 演算 | 意味 | 例 |
|------|------|-----|
| `C ÷ n` | 分割（チャンク化） | 大量出力の分割処理 |
| `C ÷ A` | 因数分解 | 結果と入力から処理を逆算 |
| `C ÷ A` | 帰属分析 | 入力Aの寄与度を測る |

### 非可換性 — `×` が `+` と決定的に違う点

```
A + B = B + A    可換（順不同）
A × B ≠ B × A    非可換（順序が結果を決める）
```

順序を入れ替えるだけで設計思想が変わる。

| 順序 | 意味 |
|------|------|
| 生成系 × 検証系 | 作ってからレビュー（量産・試行錯誤型） |
| 検証系 × 生成系 | 基準を先に決めてから作る（高品質・手戻り削減型） |

`+` だけで思考していた段階では見えなかった構造が、`×` を導入すると現れる。Agent同士を「組み合わせる」とき、**順序がそのまま設計思想を表現する** ことに気づかされる。

---

## 第6層: メタレベル（`^` `√`）

最上層は自己適用と本質抽出だ。

| 演算 | 意味 | 例 |
|------|------|-----|
| `A^n` | n回の自己適用（再帰） | Agentを生み出すAgent |
| `A^2` | 自分に自分を適用 | レビュー結果をレビューする（メタレビュー） |
| `√C` | 多数の事例から共通原理を抽出 | 複数Agentから共通設計原則を見出す |
| `log_A C` | 何段階の適用でCに到達するか | 成熟度・成長段階の測定 |

普通のAgentは `A → C` の写像だが、メタAgent（Agentを作るAgent）は `A → (B → C)` を生み出す写像、つまり **1階上の演算子** だ。

この階層構造は `+` `-` `×` `÷` だけでは表現できない。Agent開発が「Agentを作る」段階から「Agentを作る仕組みを作る」段階へ進化するとき、思考は自然に `^` の世界に入る。

---

## 改修判断フロー

ここまでの6層を統合すると、改修依頼に対する判断フローが組める。

```
改修依頼が来た
  │
  ├─ 変化するのは A / B / C のどれか
  │
  ├─ 変化は「実体レベル」か「役割レベル」か
  │    ├─ 実体レベル → A→A' / C→C' （マイナー改修・設定で吸収）
  │    └─ 役割レベル → A→E / C→D   （世代交代・型定義の更新）
  │
  └─ 右辺の状態（=0 / =C / =-C）は維持されるか
       ├─ 維持されない → 設計見直し
       └─ 維持される   → 改修実施
```

「設定ファイルを直すだけか、Agentの不変アイデンティティ部分を直すべきか」という日常の判断が、この式で機械的に切り分けられる。

## 演算性質まとめ

```
+ : 可換、並列、累積（追加は簡単）
- : 非可換、差分、判断力を要する（引き算は高次思考）
× : 三義（倍率・合成・交差）、合成は非可換、設計思想を表す
÷ : 三義（分割・因数分解・帰属）、逆算思考、原因分析
^ : 自己適用、メタレベル、再帰
√ : 共通原理の抽出、本質への遡及
```

---

## 適用ガイド — 自分のプロジェクトでの7ステップ

1. **要素を特定する** — A（入力）/ B（処理）/ C（出力）を一行で書く
2. **第1層で視点を3つ作る** — `-` 方向の改善余地を探す
3. **第2層で評価軸を選ぶ** — 右辺の状態（均衡・増幅・損失）で今の位置を判定
4. **第3層で改修種別を見極める** — `→C'` か `→D` か
5. **第4層で定義を見直す** — 型レベルで定義できているか
6. **第5層で組み合わせを設計する** — `×` の順序、`÷` の境界を明示
7. **第6層で抽象を上げる** — 共通原理（`√C`）を設計原則として書き残す

---

## おわりに

このフレームは「Agent開発の万能ツール」ではない。数式はあくまでメタファーで、厳密な代数法則が成立するわけではない。

しかし、**「足し算偏重」から抜け出すための補助線** として、演算の語彙はかなり強力だ。`+` 以外の選択肢があると気づくだけで、設計の幅が一気に広がる。

演算をさらに拡張する余地もある。

- `mod`（剰余）= 制約への適合
- `∂`（微分）= 変化の方向
- `∫`（積分）= 蓄積・履歴

必要に応じて、各自の道具箱に足してほしい。

`A + B = C` から、Agent設計の世界を旅する一助になれば幸いだ。
