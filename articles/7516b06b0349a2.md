# #35 役割の重複をどう手放したか？（実装編）  
**― 重心が移動した結果、何が“不要”になったのか ―** 🧹

前回は、エージェントネットワークの中で  
**判断の重心がどのように移動していったか** を整理した。

- 初期：生成が中心  
- 中期：品質が中心  
- 中盤：仕様が中心  
- 後半：分析が中心  
- 完成：安定性が中心  

この“重心の移動”が起きると、  
自然と **「もうこの役割いらなくない？」** という部分が出てくる。

今回は、実際にどのように重複を手放し、  
ネットワークを軽くしていったのかを、  
コード断片を交えながら解説する。

---

# 🧩 1. 重心が移動すると「重複」が浮き上がる

重複は、最初から“悪”ではない。  
むしろ初期は、複数のエージェントが似た処理を持っていても問題ない。

しかし重心が移動すると、  
**「判断の中心にいないエージェントの重複」が急に邪魔になる。**

例：  
- 生成が中心 → 生成系の重複は気にならない  
- 品質が中心 → 生成側の重複がノイズになる  
- 仕様が中心 → 分析側の重複がズレを生む  

つまり、  
**重心が移動するたびに、不要な重複が“浮き上がる”構造**になっていた。

---

# 🔍 2. 実際に重複していた部分

代表的な重複はこの3つ。

| 重複していた領域 | 何が起きていたか |
|:---|:---|
| 要件抽出 | 生成系と分析系が両方で似た処理を持っていた |
| 仕様読み取り | Spec Manager と Data Analyzer が同じWiki読み込みをしていた |
| Issue判断 | Data Analyzer と Issue Triage が似たロジックを持っていた |

これらは、重心が移動するたびに  
**「どっちが本物？」**  
という問題を生んでいた。

---

# 🧹 3. 重複を手放すために行ったこと

## ① “本物の担当”を決める  
まず、どのエージェントがその領域の“正しい担当”なのかを決めた。

例：Issue判断  
- Data Analyzer → シグナル収集  
- Issue Triage → 最終判断  

このように、  
**役割を分割し、責務を一本化する**。

---

## ② 重複コードを削除して一本化する  
実際の統合はこんな感じで進んだ。

### Before（重複していた頃）
```python
# Data Analyzer 側
def analyze_issue(issue):
    severity = estimate_severity(issue)
    return severity

# Issue Triage 側
def triage(issue):
    severity = estimate_severity(issue)
    impact = estimate_impact(issue)
    return severity, impact
```

両方が `estimate_severity` を持っていた。

### After（一本化）
```python
# 共通ロジックとして切り出し
def estimate_severity(signals):
    ...

# Data Analyzer 側
def collect_signals(issue):
    ...

# Issue Triage 側
def triage(issue):
    signals = collect_signals(issue)
    severity = estimate_severity(signals)
    impact = estimate_impact(signals)
    return severity, impact
```

**判断ロジックは Issue Triage に一本化し、  
Data Analyzer は“材料を集める側”に役割を変更した。**

---

## ③ SKILL.md の責務を整理する  
重複を手放すとき、  
SKILL.md の整理が最も効果的だった。

### Before（重複していた頃）
```
skills:
  - analyze_issue
  - triage_issue
```

### After（一本化）
```
skills:
  - collect_signals
  - triage_issue
```

**スキル名を整理するだけで、  
ネットワーク全体の責務が明確になった。**

---

# 🧠 4. 重複を手放すとネットワークが軽くなる

重複を削ると、  
ネットワーク全体が驚くほど軽くなる。

| Before | After |
|:---|:---|
| 似た処理が複数箇所に存在 | 1箇所に集約 |
| どこを直せばいいか分からない | 修正ポイントが明確 |
| エージェント同士のズレが発生 | 判断が一貫する |
| 変更の影響範囲が読めない | 影響範囲が明確 |

特に大きかったのは、  
**判断の一貫性が劇的に上がったこと。**

重複があると、  
「Aではこう判断するけどBでは違う」  
というズレが起きる。

重複を手放すことで、  
**ネットワーク全体が“同じ基準”で動くようになった。**

---

# 🌈 5. なぜこのタイミングで重複を手放せたのか？

ここが今回の一番重要なポイント。

> **重心が移動したから、重複が“不要”になった。**

- 生成が中心 → 生成の重複は許容  
- 品質が中心 → 生成の重複が邪魔  
- 仕様が中心 → 仕様の重複が致命的  
- 分析が中心 → 判断ロジックの重複が危険  
- 安定が中心 → バックアップの重複は無意味  

つまり、  
**重複を手放すタイミングは“重心が移動した瞬間”だった。**

これが自然な流れを生んだ。
