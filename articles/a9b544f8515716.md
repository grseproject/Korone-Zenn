# #49 TestRail Integrator の内部構造（実装編）  
**― “ネットワークの成果物”を外部システムに届ける仕組み ―** 🚀

TestRail Integrator は、  
エージェントネットワークの **唯一の“出口エージェント”** だった。

- Reviewer によって品質保証されたスコープを  
- JSON 形式で ingest し  
- TestRail API に送信し  
- テストケースとして公開する  

この一連の流れが成立して初めて、  
ネットワークは **「実務に使えるシステム」** になった。

しかし、最初からこの形だったわけではない。  
TestRail Integrator は **3回の構造刷新** を経て完成している。

---

# 🧩 1. 初期の問題：公開処理が不安定だった

初期の Integrator は、  
TestRail API に直接 JSON を投げるだけの単純な構造だった。

その結果、次の問題が発生した。

### ■ 問題1：JSON の形式が揺れる  
Scope Generator の出力が安定していなかったため、  
API がエラーを返すことが多かった。

### ■ 問題2：Reviewer のスコアを反映できない  
品質保証されたスコープだけを公開したいのに、  
Reviewer との連携が弱かった。

### ■ 問題3：公開後の ID 管理ができない  
TestRail に登録されたテストケースの ID を  
ネットワーク側で追跡できなかった。

### ■ 問題4：API エラー時のリトライがなかった  
ネットワークが止まる原因になった。

---

# 🧭 2. 方向性が決まった瞬間  
問題が積み上がった結果、  
TestRail Integrator の役割は明確になった。

> **「品質保証されたスコープだけを、  
>  安全に TestRail に公開する」**

つまり、  
**“出口の品質保証” と “外部連携の安定化”** が必要だった。

---

# 🧱 3. 最終構造（3モジュール構成）

TestRail Integrator は最終的に **3モジュール構成** に落ち着いた。

```
1. ingest_json（スコープの取り込み）
2. build_payload（TestRail用の整形）
3. publish（API送信）
```

構成図はこう。

```
+---------------------------+
|    TestRail Integrator    |
|---------------------------|
| ingest_json               |
| build_payload             |
| publish                   |
+-------------+-------------+
              |
              v
        [TestRail API]
```

---

# 🧠 4. SKILL.md（最終形）

```markdown
skills:
  - ingest_json: "Reviewerが承認したスコープを取り込む"
  - build_payload: "TestRail API用のペイロードを構築する"
  - publish: "TestRailにテストケースを公開する"
```

**1スキル＝1出口処理** に分割したことで、  
揺れがほぼゼロになった。

---

# 🔍 5. ingest_json（スコープ取り込み）

**背景：Scope Generator → Reviewer → Integrator の流れを安定させる必要があった。**

```python
def ingest_json(scope_json):
    if scope_json.get("approved") is not True:
        raise ValueError("Scope is not approved by Reviewer")
    return scope_json
```

ポイント：

- Reviewer の承認（score ≥ 90）が必須  
- 未承認のスコープは公開しない  
- “品質保証された成果物だけ” を扱う  

---

# 🧩 6. build_payload（TestRail用整形）

**背景：TestRail API が求める形式に合わせる必要があった。**

```python
def build_payload(scope):
    return {
        "title": scope["title"],
        "priority_id": priority_to_id(scope["priority"]),
        "custom_steps": [
            {
                "content": s["action"],
                "expected": s["expected"]
            }
            for s in scope["steps"]
        ]
    }
```

整形していた内容：

- title  
- priority  
- steps（Action/Expected）  
- TestRail の custom_steps 形式  

---

# 🚀 7. publish（API送信）

**背景：API エラー時のリトライが必要だった。**

```python
def publish(payload):
    try:
        response = testrail_api.post("add_case/123", payload)
        return response["id"]
    except Exception:
        # 1回だけリトライ
        response = testrail_api.post("add_case/123", payload)
        return response["id"]
```

ポイント：

- API エラー時は 1 回リトライ  
- 成功時は TestRail の case_id を返す  
- Orchestrator が case_id を Session Manager に保存

```
[Test Scope Generator] → MD/JSON

```

Integrator は **“品質保証された成果物だけを外部に出す”** 役割。

# 🎯 9. 成功例（ここが一番重要）

## ✔ 成功例1：TestRail 公開が完全自動化

**人手ゼロでテストケースが公開されるようになった。**

**ネットワークが止まる事故が完全に消えた。**

## ✔ 成功例3：case_id の追跡が可能に  
Integrator が返す case_id を Session Manager が保存し、  
**テストケースのライフサイクル管理が可能に。**

## ✔ 成功例4：品質保証ラインが完成  
Scope Generator（生成）  
Reviewer（品質）  
Integrator（公開）  
という **三段階の品質保証ライン** が成立。

## ✔ 成功例5：ネットワークの“出口”が安定  
出口が安定したことで、  
他のエージェントが安心して動けるようになった。
