## Claude Code開発日記 Phase 1: 初めての挑戦 🚀
### はじめに 👋

2025年8月中旬、私はClaude Codeに初めて触れました。
きっかけは、QA業務での「仕様書を効率的に読み込みたい」というシンプルな願望から。

でも、蓋を開けてみれば...まぁ大変でした 😅

### Phase 1の目標 🎯

**「QA業務で使える仕様書読み込みAgentを作る」**

これだけ聞くと簡単そうですよね？
私もそう思ってました。最初は。

### 開発環境 💻

- **ツール**: Claude Code（初体験）
- **実装言語**: Python
- **開発者スペック**: CLI未経験のVSCode民
- **時期**: 2025年8月中旬

### 補足：私のスキルセット

ターミナル？黒い画面？うん、見たことはあるよ...使ったことはないけど 🙈

### 技術選択の話 🤔

### なぜPython Agentなのか

最初に悩んだのが、どの形式でAgentを作るか。

**選択肢1: MD形式**
- シンプル
- 書きやすい
- でも...QAの仕様書読み込みには向いてない？

**選択肢2: Python Agent** ✅
- 構造化データを扱いやすい
- 処理の柔軟性が高い
- 複雑な読み込みロジックも実装できそう

結論：**Python Agentで行く！**

MD形式は手軽だけど、仕様書の複雑な構造を扱うにはPythonの方が適していると判断しました。
（この判断が正しかったかは...Phase 1を通じて検証することになります）

## Phase 1で直面したこと 😅

### 壁その1: Agentって何ができるの？ 🤷

恥ずかしながら、最初は全く分かっていませんでした。

- ファイルを読むだけ？
- 複数ファイルを扱える？
- 外部ツールを呼べる？
- どこまで自律的に動くの？
- そもそもAgentの定義って...？

**触ってみないと分からない系のやつでした。**

### 壁その2: CLI初体験 💻

`claude code` ってコマンドを打つらしい。

はい、ここで問題。
**私、コマンドライン触ったことないです。**

ターミナルは開いたことあるけど、
自分でコマンド打つのは初めて。

最初は本当に `ls` とか `cd` とかから調べました 😂

### 壁その3: 理想と現実のギャップ 📊

**期待していたこと：**
「仕様書読み込んで、良い感じにQA用のドキュメント作ってくれるんでしょ？」

**現実：**
「出力された形式が、うちのチームで使ってるものと全然違う...」

動くことは動く。
仕様書も読み込める。
でも、**「動く」と「使える」は別物**でした。

### 具体的に何が大変だったか

私たちのチームには、日々作成しているQAドキュメントのフォーマットがあります。
- 特定の項目構成
- 決まった記述スタイル
- チーム内で共有しやすい形式

Agentが最初に出力したものは、確かに「QAドキュメント」ではあったんです。
でも、それを**私たちの形式に近づける**のに、めちゃくちゃ苦労しました。

プロンプトを何度も調整して、
出力を確認して、
また調整して...

この繰り返し。

## Phase 1を振り返って 🤔

### できたこと ✅
- Claude Codeの基本的な使い方を理解
- Python Agentの作成方法を把握
- CLIの基本操作を習得（ls, cd, ありがとう）

### まだできていないこと / 課題 📋
- チームの業務フォーマットへの完全な適合
- 効率的なプロンプト設計
- Agentの機能を最大限活用できていない感覚

### Phase 1で学んだ教訓 💡

1. **いきなり完璧を目指さない**
   - まずは動かしてみる
   - 触って理解する
   - そこから改善する

2. **「動く」≠「使える」**
   - 技術的に動作することと
   - 実務で使えることは別
   - 後者の方が遥かに難しい

3. **基礎は大事**
   - CLIの基本操作
   - Agentの仕組み理解
   - これがないと先に進めない
