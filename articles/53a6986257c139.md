# #20 エージェントの役割が増えたとき、どう統合するか？  
**― “増え続ける視点” をどう扱うか ―**

前回は、エージェント同士が学習し合うことで  
チーム全体が成長していく仕組みを扱った。

しかし、成長すればするほど、  
**新しい役割が必要になる。**

- 新しい視点を追加したい  
- 新しい分析が必要になった  
- 新しい品質基準が生まれた  
- 新しい改善方針が必要になった  

つまり、  
**エージェントの役割は増え続ける。**

では、役割が増えたとき、  
どうやってチーム全体を統合すればいいのか？

---

## 🎯 役割が増えると何が起きるか？

役割が増えると、次のような問題が発生する。

- 視点が増えて Blue が混乱する  
- 調停（Mediator）の負荷が増える  
- 意見の衝突が増える  
- 改善の方向性が分散する  
- 全体の一貫性が崩れやすくなる  

つまり、  
**役割の追加は“複雑性の増加”を意味する。**

---

## 🧩 役割を整理するための「視点マップ」

まず必要なのは、  
**増えた役割を整理するための“視点マップ”を作ること。**

### ▼ 視点マップ（Zenn対応表）

| 視点 | 役割 | 目的 |
|------|------|------|
| Green | 仕様分析 | 正しさ |
| Red | 品質評価 | 安定性 |
| Meta | 改善 | 最適化 |
| Blue | 生成 | 実現 |
| White | 整形 | 読みやすさ |
| New-X | 新しい視点 | 新しい価値 |

新しい役割（New-X）は、  
既存の視点のどこに位置づけるべきかを判断する必要がある。

---

## 🔧 役割統合の基本方針

役割が増えたときの統合方針は次の3つ。

1. **視点の分類**  
2. **優先順位の付与**  
3. **調停プロセスへの組み込み**

---

## 1. 視点の分類

新しい役割が追加されたとき、  
まずは **どの視点に近いか** を分類する。

例：

- 「リスク分析」 → Red（品質）に近い  
- 「要件の深掘り」 → Green（仕様）に近い  
- 「文章の自然さ」 → White（整形）に近い  

分類できない場合は、  
**新しい視点（New-X）として追加** する。

---

## 2. 優先順位の付与

次に、  
**既存の基準モデルに優先順位を追加する。**

### ▼ 優先順位モデル（拡張版）

```
1. 仕様（Green）
2. 品質（Red）
3. 改善（Meta）
4. 新しい視点（New-X）
5. 生成（Blue）
6. 整形（White）
```

New-X は **既存の基準の中間に入る** ことが多い。

---

## 3. 調停プロセスへの組み込み

最後に、  
**調停エージェント（Mediator）が新しい視点を扱えるようにする。**

### ▼ 調停フロー（Zenn最適化図）

```
Green（仕様） → 意見A
Red（品質）   → 意見B
Meta（改善）  → 意見C
New-X（新視点）→ 意見D
Blue（生成）   → 意見E
White（整形）  → 意見F
        ↓
Mediator が優先順位に基づいて統合
        ↓
Blue が再生成
```

---

## 🧪 新しい役割を追加したときのコード例（抽象モデル）

```text
roles = ["spec", "quality", "improvement", "new_x", "generation", "formatting"]

def mediate(opinions, priority=roles):
    sorted_ops = sort_by_priority(opinions, priority)
    return integrate(sorted_ops)
```

このように、  
**新しい役割を priority に追加するだけで統合できる** 構造が理想。

---

## 🧠 役割統合がもたらした効果

- 新しい視点が自然にチームに溶け込む  
- Blue の混乱が減る  
- 調停が安定する  
- 改善が広がりやすくなる  
- 全体の一貫性が保たれる  
- チームが“拡張可能な構造”になる  

特に、  
**視点マップ → 優先順位 → 調停組み込み**  
の3段階が非常に効いた。

---

## 📚 この時期の学び

- 役割は増える前提で設計すべき  
- 視点マップが複雑性を整理する  
- 優先順位モデルは拡張可能であるべき  
- 調停プロセスは新しい視点を受け入れられる構造にする  
- 役割拡張はチームの進化を示す  

---

## 📝 一般的な技法との対応（今回）

- **Modular Architecture（モジュール化）**  
  役割を独立した視点として扱う考え方。

- **Extensible Priority Model（拡張可能な優先順位モデル）**  
  新しい基準を追加できる構造。

- **Integration Workflow（統合ワークフロー）**  
  新しい視点を既存プロセスに組み込む技法。

- **Perspective Mapping（視点マッピング）**  
  役割を分類し、位置づけるための手法。

- **Scalable Team Design（拡張可能なチーム設計）**  
  役割が増えても破綻しない構造。

→ 今回の「役割統合」は、これらの一般的な考え方をエージェント開発に応用したものです。
