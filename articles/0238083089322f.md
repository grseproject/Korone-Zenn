# #42 Quality Reviewer の内部構造（実装編）  
**― “品質の基準点”をどう作り、どうネットワーク全体を安定させたのか ―** 🔍

Test Scope Generator が“入口”なら、  
Quality Reviewer は **“基準点”** だった。

- 生成されたスコープを評価し  
- 5軸でスコアリングし  
- 90点未満なら差し戻し  
- 90点以上なら Integrator へ進める  

この **「基準点の存在」** が、  
ネットワーク全体の品質を底上げした。

しかし、最初からこの形だったわけではない。  
Quality Reviewer は、  
**3回の構造刷新と2回のスコアリング再設計** を経て完成している。

この記事では、  
**その進化の背景と、最終的に落ち着いた内部構造** をまとめる。

---

# 🧩 1. 初期の問題：レビューが揺れすぎた

初期の Reviewer は、  
「なんとなく良さそうならOK」という曖昧な基準だった。

その結果：

### ■ 問題1：差し戻し基準が揺れる  
同じスコープでも、  
- ある日はOK  
- ある日はNG  
という揺れが発生。

### ■ 問題2：Test Scope Generator が迷走  
Reviewer の基準が揺れるため、  
Generator がどこを改善すべきか分からない。

### ■ 問題3：Integrator が ingest できない  
品質が揺れると JSON の構造も揺れ、  
TestRail Integrator が失敗する。

---

# 🧭 2. 方向性が決まった瞬間  
問題が積み上がった結果、  
Reviewer の役割は明確になった。

> **「品質の基準点を作る」  
> 「揺れない判断を提供する」  
> 「ネットワーク全体の安定性を支える」**

つまり、  
**“評価者” ではなく “基準点”** にする必要があった。

---

# 🧱 3. 最終構造（5軸スコアリング）

Quality Reviewer は、  
最終的に **5軸スコアリング方式** に落ち着いた。

```
clarity（明確さ）
coverage（網羅性）
consistency（一貫性）
risk（リスク考慮）
structure（構造の整合性）
```

各軸 0〜20 点、合計 100 点。  
**90 点未満は差し戻し**。

---

# 🧠 4. SKILL.md（最終形）

```markdown
skills:
  - score_clarity: "明確さを評価する"
  - score_coverage: "網羅性を評価する"
  - score_consistency: "一貫性を評価する"
  - score_risk: "リスク考慮を評価する"
  - score_structure: "構造の整合性を評価する"
  - summarize: "総合スコアを算出し、差し戻し判断を行う"
```

**1スキル＝1軸** に分割したことで、  
揺れがほぼゼロになった。

---

# 🔍 5. 軸ごとの実装（背景＋コード）

## ① clarity（明確さ）  
**背景：曖昧な表現が多く、Reviewer が読みづらかった。**

```python
def score_clarity(md):
    penalties = 0
    if "??" in md or "TBD" in md:
        penalties += 5
    if len(md.split()) < 50:
        penalties += 3
    return max(0, 20 - penalties)
```

---

## ② coverage（網羅性）  
**背景：acceptance criteria の抜け漏れが多かった。**

```python
def score_coverage(reqs, steps):
    ac_count = len(reqs["acceptance"])
    step_count = len(steps)
    if step_count < ac_count:
        return 10
    return 20
```

---

## ③ consistency（一貫性）  
**背景：MD と JSON の内容が一致しないことがあった。**

```python
def score_consistency(md, json_data):
    if json_data["title"] not in md:
        return 15
    return 20
```

---

## ④ risk（リスク考慮）  
**背景：例外系が抜けると事故につながる。**

```python
def score_risk(steps):
    has_error = any("error" in s["action"].lower() for s in steps)
    return 20 if has_error else 15
```

---

## ⑤ structure（構造の整合性）  
**背景：MD の構造が揺れると Reviewer が読めない。**

```python
def score_structure(md):
    if "## Steps" not in md:
        return 10
    return 20
```

# 🔄 6. 総合スコアと差し戻し判断

```python
def summarize(scores):
    total = sum(scores.values())
    return {
        "score": total,
        "status": "ok" if total >= 90 else "reject",
        "axes": scores
    }
```

# 🎯 8. 成功例（ここが一番重要）

## ✔ 成功例1：差し戻し率が激減  
5軸スコアリング導入後、  
**差し戻し率が 40% → 8% に低下**。

## ✔ 成功例2：TestRail 公開が安定  
JSON の揺れが消え、  
Integrator の成功率が 100% に。

## ✔ 成功例3：Generator の改善が高速化  
Reviewer の軸が明確なので、  
Generator がどこを改善すべきか一目で分かる。

## ✔ 成功例4：ネットワーク全体の品質が底上げ  
Reviewer が“基準点”になったことで、  
他のエージェントの判断が揺れなくなった。

---

# 🌈 9. まとめ  
Quality Reviewer は、  
**ネットワーク全体の品質を支える“基準点エージェント”**。

- 5軸スコアリング  
- 差し戻し判断  
- Generator との連携  
- Integrator との接続  
- 揺れない基準の提供  

これらが揃ったことで、  
ネットワーク全体が安定し、  
“品質の土台” が完成した。
