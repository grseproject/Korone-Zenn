# #32 視点どうしの“会話”をデザインする（実装編）  
**― 抽象で語ってきた視点が、実装の中でどう動いていたのか ―** 🔍

今回からは、実際に構築した14エージェントの開発ログを、“実装編”として読み解いていく。

視点は概念ではなく、  
**実際のエージェントたちが担っていた“役割そのもの”** だった。

---

## 🔄 視点は“会話”することで機能する

視点は単体では完結しない。  
複数の視点が互いに情報を渡し、補完し、時に衝突しながら、  
**チーム全体の判断をつくっていく。**

この“視点どうしの会話”は、  
11月〜2月のエージェント開発でそのまま実装されていた。

---

## 🧭 実装における「視点の会話」の代表例

ここでは、実際のエージェント間連携を  
“視点の会話”として読み解いていく。

---

### 1. Parallel Advisor × Data Analyzer  
**（思考視点 × 分析視点）**

Parallel Advisor は「依存関係」「並列化可能性」を判断する“思考視点”。  
Data Analyzer は「情報整理」「KPI」「Issue判断」を担う“分析視点”。

この2つは、1月の開発で頻繁に会話していた。

| 視点 | 実装での役割 |
|:---|:---|
| 思考視点 | タスクをどう分割するか、どこまで並列化できるかを判断 |
| 分析視点 | タスクの内容・粒度・依存情報を整理し、思考視点に渡す |

この連携により、  
**「並列化できるタスク」と「できないタスク」が自動で分類される仕組み** が生まれた。

---

### 2. Code Reviewer × Test Scope Generator  
**（品質視点 × 生成視点）**

Test Scope Generator が作ったスコープは、  
必ず Code Reviewer（品質視点）を通過する。

| 視点 | 実装での役割 |
|:---|:---|
| 生成視点 | テストスコープを作る |
| 品質視点 | 100点満点の5軸スコアで品質を評価し、基準に満たない場合は差し戻す |

これはまさに  
**視点どうしの“レビュー会話”**。

12月〜1月の開発で、  
この会話が自動化され、  
**「品質ゲート」** として機能するようになった。

---

### 3. Spec Manager × Wiki  
**（仕様視点 × 情報源）**

Spec Manager は“仕様視点”を担うエージェント。  
Wiki は“仕様の真実”が置かれる場所。

Spec Manager は Wiki と会話しながら、  
- 差分検出  
- Git同期  
- 更新  
- QA回答  

を行う。

| 視点 | 実装での役割 |
|:---|:---|
| 仕様視点 | 仕様の正しさ・最新性を担保する |
| 情報源 | 仕様の“真実”を保持する場所 |

この会話があったからこそ、  
**仕様のズレが自動で検出される仕組み** が成立した。

---

### 4. Session Manager × 全エージェント  
**（記録視点 × 全視点）**

Session Manager は“記録視点”。  
全エージェントの活動を30分ごとにバックアップし、  
クラッシュ時の復旧を支えた。

| 視点 | 実装での役割 |
|:---|:---|
| 記録視点 | 進行中の作業を安全に保存する |
| 各視点 | 自分の判断や生成物を Session Manager に渡す |

この会話があったから、  
**作業の安全性と継続性が保証された**。

---

## 🌈 視点どうしの会話が生んだ“チームのハーモニー”

11月〜2月の開発を振り返ると、  
視点どうしの会話が自然に生まれていた。

- 思考視点が分析視点に相談する  
- 生成視点が品質視点にチェックされる  
- 仕様視点が情報源と同期する  
- 記録視点が全視点を支える  

これはまさに、  
**視点シリーズで語ってきた抽象モデルが、実装の中でそのまま動いていた瞬間** だった。

---

## 📚 この時期の学び

| 学び | 内容 |
|:---|:---|
| 視点は単体では機能しない | 会話することで初めて判断が生まれる |
| エージェントは視点の具現化 | それぞれが1つの視点を担当していた |
| 会話の設計がチームの安定を生む | 連携が自然に起きる構造が重要 |
| 抽象と具体は往復する | 視点モデルが実装を支え、実装が視点を育てる |
