# #45 Data Analyzer の内部構造（実装編）  
**― “多スキル型エージェント”がネットワークの判断をどう支えたか ―** 📊

Data Analyzer は、  
あなたのエージェントネットワークの中で **最も複雑で、最も影響範囲が広いエージェント** だった。

- Excel  
- Wiki  
- KPI  
- report  
- triage（材料集め）  

これらを **1つのエージェントで扱う** という構造は、  
ネットワークの中で唯一の存在だった。

しかし、最初からこの形だったわけではない。  
Data Analyzer は **6回の構造刷新** を経て、  
最終的に “多スキル型の情報ハブ” として完成した。

---

# 🧩 1. 初期の問題：情報が分散しすぎていた

初期のネットワークでは、  
情報が複数のエージェントに分散していた。

### ■ 問題1：Excel解析が別エージェント  
### ■ 問題2：Wiki監視が別エージェント  
### ■ 問題3：KPI算出が別エージェント  
### ■ 問題4：Issue triage の材料が揃わない  
### ■ 問題5：レポート生成が孤立していた  

その結果、  
**「どの情報が正しいのか」** が分からなくなり、  
判断の一貫性が失われていた。

---

# 🧭 2. 方向性が決まった瞬間  
問題が積み上がった結果、  
Data Analyzer の役割は明確になった。

> **「情報を集めるエージェント」ではなく  
>  “情報を統合するエージェント” にする。**

つまり、  
**“情報の入口” ではなく “情報のハブ”** にする必要があった。

---

# 🧱 3. 最終構造（多スキル構成）

Data Analyzer は最終的に **5スキル構成** に落ち着いた。

```
excel
wiki
kpi
report
triage（材料集め）
```

構成図はこう。

```
+----------------------+
|    Data Analyzer     |
|----------------------|
| excel   | wiki       |
| kpi     | report     |
| triage  |            |
+---------+------------+
          |
          v
   [Issue Triage]
```

---

# 🧠 4. SKILL.md（最終形）

```markdown
skills:
  - excel: "Excelデータを解析し、統計情報を抽出する"
  - wiki: "Wikiを監視し、仕様の変化を検出する"
  - kpi: "テスト・開発のKPIを算出する"
  - report: "分析結果をレポート形式にまとめる"
  - triage: "Issue判断のための材料を収集する"
```

**1スキル＝1情報源** に分割したことで、  
揺れがほぼゼロになった。

---

# 🔍 5. excel（Excel解析）

**背景：テスト結果やログが Excel で渡されることが多かった。**

```python
def excel(path):
    df = read_excel(path)
    return {
        "rows": len(df),
        "columns": list(df.columns),
        "summary": df.describe().to_dict()
    }
```

抽出していた情報：

- 行数  
- 列名  
- 統計情報  
- 欠損値  

---

# 📘 6. wiki（Wiki監視）

**背景：Spec Manager と連携し、仕様の変化を検出する必要があった。**

```python
def wiki(page):
    raw = wiki_api.get(page)
    parsed = parse_yaml(raw)
    return parsed
```

Spec Manager との違い：

- Spec Manager → “真実性を守る”  
- Data Analyzer → “変化を検出する”  

---

# 📈 7. kpi（KPI算出）

**背景：テストの進捗や品質を数値化する必要があった。**

```python
def kpi(data):
    return {
        "pass_rate": data["passed"] / data["total"],
        "fail_rate": data["failed"] / data["total"],
        "coverage": len(data["executed"]) / len(data["planned"])
    }
```

KPI は Issue triage の判断材料にも使われた。

---

# 📝 8. report（レポート生成）

**背景：Analyzer の出力を人間が読める形にする必要があった。**

```python
def report(analysis):
    md = "# Analysis Report\n\n"
    for key, value in analysis.items():
        md += f"- **{key}:** {value}\n"
    return md
```

---

# 🔎 9. triage（Issue判断の材料集め）

**背景：Issue Triage が判断するための“材料”を集める必要があった。**

```python
def triage(issue):
    signals = {
        "labels": issue.get("labels", []),
        "comments": len(issue.get("comments", [])),
        "priority": issue.get("priority", "medium"),
        "history": issue.get("history", [])
    }
    return signals
```

ここで判断はしない。  
**判断は Issue Triage の役割。**

Data Analyzer はあくまで “材料集め”。

---

# 🕸️ 10. 他エージェントとの関係性（構成図）

```
                 +------------------+
                 |  Spec Manager    |
                 | (真実性の維持)   |
                 +--------+---------+
                          |
                          v
+------------------+     +------------------+
| Data Analyzer    | --> | Issue Triage     |
| (情報のハブ)     |     | (判断の専門家)   |
+------------------+     +------------------+
```

- Spec Manager → “真実性”  
- Data Analyzer → “情報の統合”  
- Issue Triage → “判断”  

という **三位一体構造** が完成した。

---

# 🎯 11. 成功例（ここが一番重要）

## ✔ 成功例1：Issue判断の精度が劇的に向上  
Data Analyzer が材料を揃え、  
Issue Triage が判断する構造にしたことで、  
**判断の揺れがほぼゼロに。**

## ✔ 成功例2：仕様の変化が即座に検出  
Wikiスキルが Spec Manager と連携し、  
**仕様のズレがその日のうちに検出されるように。**

## ✔ 成功例3：KPI が安定して算出される  
KPI が安定したことで、  
テスト進捗の可視化が容易に。

## ✔ 成功例4：レポート生成が自動化  
report スキルにより、  
**分析 → レポート → 公開** が自動化。

## ✔ 成功例5：ネットワーク全体の判断が安定  
Data Analyzer が“情報のハブ”になったことで、  
他のエージェントが迷わなくなった。

---

# 🌈 12. まとめ  
Data Analyzer は、  
**ネットワーク全体の判断を支える“情報のハブエージェント”**。

- Excel解析  
- Wiki監視  
- KPI算出  
- レポート生成  
- Issue triage の材料集め  

これらが揃ったことで、  
ネットワークは **揺れない判断基盤** を手に入れた。
