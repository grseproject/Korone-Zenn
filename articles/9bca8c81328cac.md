# #46 Parallel Advisor の内部構造（実装編）  
**― “依存関係の判断”を外部化し、ネットワーク全体のスループットを上げた仕組み ―** ⚡

Parallel Advisor は、  
あなたのエージェントネットワークの中で **最も“影響範囲が広い判断エージェント”** だった。

- Orchestrator の負荷を肩代わりし  
- タスクの依存関係を解析し  
- 並列化できるタスクを分類し  
- 安全な実行順序を提案し  
- ネットワーク全体のスループットを向上させる  

つまり、  
**“ネットワークのテンポを決めるエージェント”**。

しかし、最初からこの形だったわけではない。  
Parallel Advisor は **3回の構造刷新** を経て完成している。

---

# 🧩 1. 初期の問題：Orchestrator が過負荷だった

初期のネットワークでは、  
Orchestrator がすべての判断を抱えていた。

### ■ 問題1：依存関係の判断が重すぎる  
- A → B → C のような直列構造  
- A と C は独立なのに直列で実行される  
- 無駄な待ち時間が発生

### ■ 問題2：並列化の判断が揺れる  
Orchestrator が毎回違う判断をするため、  
**実行順序が安定しない。**

### ■ 問題3：ネットワーク全体のテンポが遅い  
特に Data Analyzer や Spec Manager の処理が重く、  
直列実行だと全体が詰まる。

### ■ 問題4：Orchestrator のコードが肥大化  
依存関係解析が Orchestrator に混ざり、  
**制御と判断が混在する“アンチパターン”** に。

---

# 🧭 2. 方向性が決まった瞬間  
問題が積み上がった結果、  
Parallel Advisor の役割は明確になった。

> **「依存関係の判断を Orchestrator から外部化する」**  
> **「並列化できるタスクを分類する専門家にする」**

つまり、  
**“制御” と “判断” を分離する** 必要があった。

---

# 🧱 3. 最終構造（3モジュール構成）

Parallel Advisor は最終的に **3モジュール構成** に落ち着いた。

```
1. detect_dependencies（依存関係の検出）
2. classify_tasks（タスク分類）
3. propose_parallel_plan（並列化計画の生成）
```

構成図はこう。

```
+---------------------------+
|     Parallel Advisor      |
|---------------------------|
| detect_dependencies       |
| classify_tasks            |
| propose_parallel_plan     |
+-------------+-------------+
              |
              v
        [Orchestrator]
```

# 🔍 5. detect_dependencies（依存関係の検出）

**背景：依存関係を誤るとネットワークが壊れる。**

```python
def detect_dependencies(tasks):
    deps = {}
    for t in tasks:
        deps[t["id"]] = t.get("depends_on", [])
    return deps
```

検出していた依存関係：

- ファイル依存  
- Wiki依存  
- Analyzer依存  
- Scope → Reviewer → Integrator の直列依存  
- Spec Manager → Scope Generator の依存  

---

# 🧩 6. classify_tasks（タスク分類）

**背景：依存関係を見て、並列化できるタスクを分類する必要があった。**

```python
def classify_tasks(tasks, deps):
    parallel = []
    blocked = []

    for t in tasks:
        if deps[t["id"]]:
            blocked.append(t)
        else:
            parallel.append(t)

    return parallel, blocked
```

分類結果：

- **parallel** → すぐ実行してよい  
- **blocked** → 依存タスクが終わるまで待つ  

---

# ⚡ 7. propose_parallel_plan（並列化計画の生成）

**背景：Orchestrator が扱いやすい形式に変換する必要があった。**

```python
def propose_parallel_plan(parallel, blocked):
    return {
        "parallelizable": [t["id"] for t in parallel],
        "blocked": [t["id"] for t in blocked]
    }
```

出力例：

```json
{
  "parallelizable": ["analyze_excel", "fetch_wiki"],
  "blocked": ["generate_scope"]
}
```

# 🕸️ 8. Orchestrator との関係性（構成図）

```
[Tasks]
   |
   v
[Parallel Advisor]
   |
   v
[Parallel Plan]
   |
   v
[Orchestrator]
   |
   +--> parallelizable → 並列実行
   |
   +--> blocked → 直列実行
```

Parallel Advisor が判断し、  
Orchestrator は **“実行だけ”** に集中できるようになった。
