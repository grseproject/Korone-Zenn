# #39 アップデート版として戻ってきた役割とは？（実装編）  
**― 一度“卒業”した役割が、どう再登場し、ネットワークを強化したのか ―** 🔁

前回は、役割が“卒業”したときに  
ネットワークがどう軽くなったかを整理した。

しかし、卒業は「消える」だけではない。  
**一度消えた役割が、アップデート版として戻ってくることもある。**

今回は、実際に戻ってきた役割と、  
そのときの **構成図・関係性の変化** を中心にまとめる。

---

# 🧩 1. 戻ってきた役割の代表例

実際に“アップデート版として戻ってきた”のはこの3つ。

| 戻ってきた役割 | どう変わったか |
|:---|:---|
| Issue判断 | Data Analyzer → Issue Triage に完全分離して再登場 |
| Wiki差分検出 | Spec Manager の内部スキルとして再構築 |
| 並列化判断 | Parallel Advisor として独立エージェント化 |

これらは「古い形では不要だった」が、  
**新しい構造に合わせて再登場した**という点が重要。

---

# 🕸️ 2. 戻ってきたときの“構成図”がこちら

## 🔁 Issue判断の再登場（Before → After）

### Before（初期）
```
Data Analyzer
 ├─ analyze_issue
 └─ estimate_severity
```

### After（再登場）
```
Data Analyzer
 └─ collect_signals

Issue Triage
 ├─ estimate_severity
 ├─ estimate_impact
 └─ recommend_action
```

**判断ロジックが Issue Triage に移動し、  
Data Analyzer は“材料集め”に専念できるようになった。**

---

## 🔁 Wiki差分検出の再登場（Before → After）

### Before（初期）
```
Data Analyzer
 └─ read_wiki
```

### After（再登場）
```
Spec Manager
 ├─ fetch_wiki
 ├─ detect_diff
 └─ update_wiki
```

**Wiki関連の責務が Spec Manager に集約され、  
仕様管理の一貫性が大幅に向上した。**

---

## 🔁 並列化判断の再登場（Before → After）

### Before（初期）
```
Orchestrator
 └─ decide_parallel()
```

### After（再登場）
```
Parallel Advisor
 ├─ detect_dependencies
 ├─ classify_tasks
 └─ propose_parallel_plan
```

**並列化判断が独立エージェント化し、  
複雑な依存関係でも安定して扱えるようになった。**

---

# 🧠 3. なぜ“戻ってきた”のか？（因果関係）

戻ってきた理由はすべて同じ。

> **ネットワークが成熟したことで、  
>  その役割を“専門家”として切り出す必要が生まれたから。**

- Issue判断 → 判断の精度が求められた  
- Wiki差分 → 仕様の真実性が重要になった  
- 並列化判断 → タスクが増えて依存関係が複雑化した  

つまり、  
**戻ってきたのではなく、“必要になったから復活した”。**

---

# 🎯 4. 再登場がもたらした“成功例”

## ✔ 成功例1：Issue判断の精度が安定  
Issue Triage が独立したことで、  
判断の揺れがほぼゼロになった。

```json
{
  "severity": "high",
  "impact": "low",
  "recommendation": "investigate"
}
```

---

## ✔ 成功例2：仕様のズレが即検出  
Spec Manager に集約したことで、  
Wikiの更新漏れがほぼ消えた。

```json
[
  {
    "field": "timeout",
    "local": 30,
    "wiki": 20
  }
]
```

---

## ✔ 成功例3：並列化が安全に  
Parallel Advisor の導入で、  
依存関係の誤判定がなくなった。

```json
{
  "parallelizable": ["taskA", "taskC"],
  "blocked": ["taskB"]
}
```

---

# 🌈 5. 再登場の本質は「専門家としての復活」

卒業 → 消える  
ではなく、

卒業 → 必要な形に再構築されて戻る

という流れ。

| フェーズ | 状態 |
|:---|:---|
| 初期 | なんでも屋 |
| 中期 | 重複が邪魔になる |
| 卒業 | 一度消える |
| 再登場 | 専門家として復活 |
| 完成 | ネットワークの要になる |

この循環が、  
**ネットワークの強さを底上げした。**
