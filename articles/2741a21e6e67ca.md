# #48 Orchestrator の内部構造（実装編）  
**― “ネットワークの心臓部”はどう設計され、どう進化したのか ―** 🎛️

Orchestrator は、エージェントネットワーク全体の **フロー制御・実行順序・状態管理** を担う中心的存在だった。

- 各エージェントの呼び出し  
- 並列化計画の適用  
- Session Manager との連携  
- エラー時のリカバリ  
- 全体のテンポ管理  

これらを一手に引き受けるため、  
Orchestrator は **5回の構造刷新** を経て、最終的に“心臓部”として完成した。

---

# 🧩 1. 初期の問題：Orchestrator が“なんでも屋”だった

初期の Orchestrator は、  
制御・判断・状態管理・エラー処理をすべて抱えていた。

その結果、次の問題が発生した。

### ■ 問題1：コードが肥大化  
依存関係解析、並列化判断、状態管理が混在し、  
**1,000行を超える巨大モジュール**になった。

### ■ 問題2：判断が揺れる  
- 並列化するか  
- どのエージェントを呼ぶか  
- どの順序で実行するか  
これらが毎回揺れた。

### ■ 問題3：クラッシュ時に復旧できない  
状態管理が内部に埋め込まれていたため、  
Session Manager と連携できなかった。

### ■ 問題4：ネットワーク全体のテンポが不安定  
重い処理が直列で実行され、  
全体のスループットが低下。

---

# 🧭 2. 方向性が決まった瞬間  
問題が積み上がった結果、  
Orchestrator の役割は明確になった。

> **「判断を外部化し、制御に専念する」**  
> **「状態管理を Session Manager に委譲する」**  
> **「並列化判断を Parallel Advisor に任せる」**

つまり、  
**“制御の専門家”として再構築する必要があった。**

---

# 🧱 3. 最終構造（4モジュール構成）

Orchestrator は最終的に **4モジュール構成** に落ち着いた。

```
1. load_tasks（タスク読み込み）
2. apply_parallel_plan（並列化計画の適用）
3. execute_tasks（エージェント呼び出し）
4. update_state（状態更新）
```

構成図はこう。

```
+---------------------------+
|        Orchestrator       |
|---------------------------|
| load_tasks                |
| apply_parallel_plan       |
| execute_tasks             |
| update_state              |
+-------------+-------------+
              |
              v
   [Parallel Advisor]   [Session Manager]
```

# 🔍 5. load_tasks（タスク読み込み）

```python
def load_tasks(ticket):
    return [
        {"id": "fetch_spec", "agent": "Spec Manager"},
        {"id": "analyze_data", "agent": "Data Analyzer"},
        {"id": "generate_scope", "agent": "Test Scope Generator"},
        {"id": "review_scope", "agent": "Quality Reviewer"},
        {"id": "publish", "agent": "TestRail Integrator"}
    ]
```

タスクは **“ネットワークの標準フロー”** として定義されている。

---

# ⚡ 6. apply_parallel_plan（並列化計画の適用）

Parallel Advisor の判断を適用する。

```python
def apply_parallel_plan(tasks, plan):
    parallel = [t for t in tasks if t["id"] in plan["parallelizable"]]
    blocked = [t for t in tasks if t["id"] in plan["blocked"]]
    return parallel, blocked
```

Parallel Advisor が  
**“判断”**  
Orchestrator が  
**“実行”**  
という役割分担が成立した。

---

# ▶️ 7. execute_tasks（エージェント呼び出し）

```python
def execute_tasks(parallel, blocked):
    results = {}

    # 並列実行
    for t in parallel:
        results[t["id"]] = call_agent(t["agent"])

    # 直列実行
    for t in blocked:
        results[t["id"]] = call_agent(t["agent"])

    return results
```

ポイント：

- 並列化は Parallel Advisor の判断に従う  
- Orchestrator は“実行だけ”に集中  
- エラー時は Session Manager に委譲  

---

# 🔄 8. update_state（状態更新）

Session Manager と連携して状態を保存する。

```python
def update_state(state, results):
    state["results"].update(results)
    state["timestamp"] = now()
    return state
```

更新された状態は Session Manager に渡され、  
**バックアップ → 整合性チェック → 保存**  
の流れに入る。
