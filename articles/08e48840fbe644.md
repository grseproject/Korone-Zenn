# #8 Vault構造が本領発揮し始める  
**― Intent / Context / Agents / Result が“システム”を支える ―**

前回は、単一責任の原則に本気で向き合い、  
エージェント同士の境界線を明確にした話を書いた。

今回は、その次のフェーズ。  
**Vault構造が本格的に効き始めた瞬間**について。

エージェントが増え始めると、  
「どこに何を書く？」「何を渡す？」「どこまで覚えさせる？」  
という問題が一気に増える。

その混乱を救ってくれたのが、  
Intent / Context / Agents / Result の **Vault構造**だった。

---

## 🗂️ Vault構造とは何か？

私が使っている Vault構造は、次の4つで成り立っている。

- **Intent**：やりたいこと（目的）  
- **Context**：材料（仕様・ログ・前提）  
- **Agents**：実際に動くエージェント  
- **Result**：最終的な成果物  

この4つを分けるだけで、  
エージェント開発の混乱が一気に整理される。

---

## 🎨 図解：Vault構造の流れ（軽量版）

Intent → Context → Agents → Result  
🎯 → 📚 → 🤖🤖🤖 → 📦

- Intent：何をしたい？  
- Context：そのために必要な材料は？  
- Agents：誰（どのエージェント）が動く？  
- Result：最終的に何を得る？  

この流れを固定したことで、  
**エージェントが増えても破綻しない土台**ができた。

---

## 🔍 Intent と Context の切り分けが特に重要だった

最初は Intent と Context が混ざりがちだった。

- 「スコープを作りたい（Intent）」  
- 「仕様書はこれ（Context）」  
- 「この部分は重要（Context）」  
- 「こういう粒度で作ってほしい（Intent）」  

混ざると、エージェントが混乱する。

そこで、次のルールを作った。

### Intent  
- 目的  
- 期待するアウトプット  
- 粒度や方針  

### Context  
- 仕様書  
- 画面キャプチャ  
- 既存のテストケース  
- 制約条件  

これを分けるだけで、  
エージェントの出力が安定し始めた。

---

## 🤖 Agents の配置パターンが見えてきた

エージェントが3体、4体と増えてくると、  
「どの順番で動かす？」  
「どこでバトンを渡す？」  
という問題が出てくる。

Vault構造を使うと、自然にパターンが見えてきた。

### パターン1：直列  
ScopeGen → Reviewer → Formatter

### パターン2：並列  
SpecChecker  
NamingChecker  
ImpactAnalyzer  
（結果をまとめる Aggregator が最後に動く）

### パターン3：ハイブリッド  
直列と並列を組み合わせる。

Vault構造があると、  
**どのパターンでも破綻しない**。

---

## 📦 Result の形式を固定したら、さらに安定した

Result の形式を固定したことで、  
後続の処理が壊れなくなった。

例：

- Markdown  
- JSON  
- TestRail用の表形式  
- 日次ノート形式  

「Resultは必ずこの形式で返す」  
というルールを作ると、  
エージェント同士の連携が一気にスムーズになった。

---

## ✨ Vault構造が効き始めた瞬間の実感

- エージェントが増えても混乱しない  
- Context の追加・削除が簡単  
- Intent が明確なので出力がブレない  
- Result の形式が固定されているので後処理が楽  
- 新しいエージェントを追加しやすい  

特に、**仕様変更検知エージェント**を追加したとき、  
Vault構造のありがたみを強く感じた。

---

## 📚 この時期の学び

- Intent と Context を分けるだけで出力の安定度が上がる  
- Agents の配置パターンは Vault構造があると自然に決まる  
- Result の形式を固定すると連携が壊れない  
- Vault構造は“エージェントが増える未来”に耐えるための基盤
