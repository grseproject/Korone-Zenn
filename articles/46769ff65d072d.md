# #36 広がりとまとまりをどう両立したか？（実装編）  
**― 多スキル化で“広がった”エージェントを、どうやって破綻させずにまとめたのか ―** 🧭

前回は、判断の重心が移動する中で  
**重複を手放す必要が生まれた理由**を整理した。

今回はその続きとして、  
**役割を整理した結果、どう成功につながったのか**  
を具体的に見ていく。

特にポイントになるのは、

- Data Analyzer の多スキル化  
- Spec Manager の機能拡張  
- Test Scope Generator の出力拡張  

これらが“広がり”を生んだ一方で、  
**ネットワーク全体が破綻しないようにどう“まとめた”のか**。

そしてその結果、  
**何ができるようになったのか**。

---

# 🧩 1. 広がりの原因：多スキル化が止まらなかった

ネットワークが成熟してくると、  
「この処理もできると便利だよね」  
「この判断も自動化したい」  
という要求が自然に増えていく。

特に Data Analyzer は、  
次々とスキルが追加されていった。

```yaml
skills:
  - excel
  - wiki
  - report
  - kpi
  - triage
```

このままでは **“便利だけど巨大で扱いづらいエージェント”** になる。

広がりは必要だったが、  
**広がりっぱなしでは成功しない。**

---

# 🧭 2. まとまりを作るためにやったこと

## ① スキルごとに“責務の境界”を引いた  
多スキル化したエージェントは、  
**「何をやるか」ではなく「どこまでやるか」** を決める必要がある。

Data Analyzer の場合：

- excel → 数値処理  
- wiki → 差分監視  
- report → テスト結果の集計  
- kpi → KPIの算出  
- triage → Issue判断の材料集め  

ここで重要なのは、  
**triage は“判断”ではなく“材料集め”に限定したこと。**

判断は Issue Triage に任せる。

これにより、  
**広がりながらも破綻しない構造**ができた。

---

## ② スキルを“モジュール化”して独立させた  
Data Analyzer の内部は、  
スキルごとに完全に分離した。

```python
# excel.py
def analyze_excel(path):
    ...

# wiki.py
def monitor_wiki(page):
    ...

# triage.py
def collect_signals(issue):
    ...
```

これにより、

- どのスキルが壊れても他に影響しない  
- 修正が局所化される  
- 新スキルを追加しても安全  

という **“広がりに強い構造”** が生まれた。

---

## ③ SKILL.md を“インデックス”として使った  
SKILL.md は単なる定義ファイルではなく、  
**エージェントの目次**として機能した。

```markdown
skills:
  - excel: "Excel-based analysis"
  - wiki: "Wiki monitoring and diff extraction"
  - triage: "Issue signal collection"
```

これにより、

- 何ができるかが一目で分かる  
- どこにコードがあるか迷わない  
- 新メンバーでもすぐ理解できる  

という **“まとまり”** が生まれた。

---

# 🎯 3. 広がりとまとまりを両立した結果、何ができるようになったのか？

ここが一番大事。  
**成功例につながっていないと意味がない。**

## ✔ 成功例1：Issue判断の精度が上がった  
Data Analyzer が材料を集め、  
Issue Triage が判断する構造にしたことで、

- 判断の一貫性  
- 精度  
- 再現性  

が大幅に向上した。

実際の triage 出力はこうなった。

```json
{
  "severity": "high",
  "impact": "medium",
  "recommendation": "investigate immediately"
}
```

以前は判断が揺れていたが、  
**今は常に同じ基準で判断できる。**

---

## ✔ 成功例2：Wiki差分の検出が安定した  
Spec Manager の wiki スキルを独立させたことで、  
差分検出が安定し、  
**仕様のズレが自動で見つかるようになった。**

```json
[
  {
    "field": "timeout",
    "local": 30,
    "wiki": 20
  }
]
```

これにより、  
**仕様の更新漏れがほぼゼロになった。**

---

## ✔ 成功例3：テストスコープの公開が自動化された  
Test Scope Generator の出力を  
- MD  
- JSON  

の2形式にしたことで、  
TestRail Integrator や Wiki公開が自動化された。

```json
{
  "title": "Login Test",
  "steps": [...],
  "priority": "P1"
}
```

これにより、  
**テスト公開までの時間が劇的に短縮された。**

---

# 🌈 4. まとめ：広がりとまとまりはセットで成功する

| 広がり | まとまり |
|:---|:---|
| 多スキル化 | 責務の境界を引く |
| 機能追加 | モジュール化 |
| 情報量増加 | SKILL.md で整理 |
| 判断の複雑化 | 判断の担当を一本化 |

この2つが揃ったとき、  
**エージェントネットワークは“破綻せずに強くなる”。**

そしてその結果、  
- Issue判断の精度向上  
- 仕様管理の安定  
- テスト公開の高速化  
という **具体的な成功例** が生まれた。
