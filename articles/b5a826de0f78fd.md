## はじめに

テスト工数の見積もりは、QAエンジニアにとって永遠の課題です。「この機能改修、何時間かかる？」という問いに対し、経験と勘に頼った回答をしていませんか？

本記事では、Claude CodeのAgent機能を活用して、**構造化されたルールに基づく工数見積もりシステム**を構築した事例を紹介します。

## 課題：従来の見積もり方法の限界

### よくある問題

1. **属人化**: ベテランと新人で見積もり精度に差がある
2. **一点見積もり**: 「3時間」という単一値では不確実性を表現できない
3. **影響範囲の見落とし**: 機能間の連動や副作用を考慮しにくい
4. **再現性がない**: 同じ条件でも人によって異なる結果

### 目指す姿

```
入力: 機能レベル + 変更タイプ + 制約条件
出力: 設計 1.5h〜3.0h / 実行 3.0h〜6.0h（範囲で提示）
```

## アプローチ：DITA構造化ルール

### DITAとは

DITA（Darwin Information Typing Architecture）は、技術文書の構造化手法です。この考え方を工数見積もりに応用しました。

### 核となる概念

| 概念 | 説明 |
|------|------|
| **階層構造** | Account → Campaign → Group → Item の4階層 |
| **機能連動** | Create実装時はEdit確認も必要（Same-level linkage） |
| **変更タイプ** | UI/課金/データモデル/連携/ロジック |
| **範囲見積もり** | 最小〜最大のレンジで提示 |

### 変更タイプと副作用リスク

```json
{
  "changeTypes": {
    "UI": {
      "sideEffectRisk": "low",
      "scope": ["display"],
      "multiplier": 1.0
    },
    "Billing": {
      "sideEffectRisk": "high",
      "scope": ["calculation", "validation", "report"],
      "multiplier": 1.5
    },
    "DataModel": {
      "sideEffectRisk": "medium",
      "scope": ["report", "export", "api"],
      "multiplier": 1.3
    }
  }
}
```

### ルールファイル（dita_rules.json）

```json
{
  "version": "1.0.0",
  "hierarchy": {
    "levels": {
      "Account": {
        "functions": ["Create", "Edit", "List", "Settings"],
        "linkage": {
          "Create": ["Edit"]
        }
      },
      "Campaign": {
        "functions": ["Create", "Edit", "List", "Copy"],
        "linkage": {
          "Create": ["Edit", "List"]
        }
      }
    }
  },
  "estimationScope": {
    "philosophy": "min-max range, not point estimate",
    "minimum": "direct impact only",
    "maximum": "include side effects and linkage"
  }
}
```

### 計算ロジック（Python）

```python
def calculate_estimation_range(
    self,
    base_design: float,
    base_execution: float,
    change_type: str = "UI",
    include_linkage: bool = True
) -> dict:
    """DITAルールに基づく見積もり範囲を計算"""

    # 変更タイプから副作用リスクを取得
    change_config = self.dita_rules["changeTypes"].get(change_type, {})
    side_effect_risk = change_config.get("sideEffectRisk", "low")

    # 最小値（直接影響のみ）
    min_design = base_design
    min_execution = base_execution

    # リスクに応じた係数
    risk_multipliers = {
        "low": 1.0,      # UI変更 - 拡張なし
        "medium": 1.3,   # データモデル - 30%拡張
        "high": 1.5      # 課金/ロジック - 50%拡張
    }
    multiplier = risk_multipliers.get(side_effect_risk, 1.0)

    # 最大値を計算
    max_design = base_design * multiplier
    max_execution = base_execution * multiplier

    # 機能連動がある場合は追加
    if include_linkage:
        max_design *= 1.2
        max_execution *= 1.2

    return {
        "minimum": {"design": min_design, "execution": min_execution},
        "maximum": {"design": max_design, "execution": max_execution},
        "format": f"設計: {min_design:.1f}h〜{max_design:.1f}h / 実行: {min_execution:.1f}h〜{max_execution:.1f}h"
    }
```

## 学び・Tips

### 1. 範囲見積もりの価値

一点見積もり（「3時間」）ではなく範囲（「2h〜5h」）で提示することで：
- 不確実性を正直に伝えられる
- リスクバッファの根拠を示せる
- ステークホルダーとの期待値調整が容易に

### 2. ルールの外部化

ロジックをコードにハードコードせず、JSONファイルで外部化することで：
- 非エンジニアでもルール調整可能
- バージョン管理でルール変更履歴を追跡
- A/Bテストでルール効果を検証可能

### 3. Claude Code Agentの使い所

Agentが有効なケース：
- ✅ 構造化されたルールに基づく判断
- ✅ 複数ファイル/ツールを組み合わせる処理
- ✅ 繰り返し実行される定型タスク

Agentに向かないケース：
- ❌ 完全に創造的な判断が必要なタスク
- ❌ 1回限りの単純な作業

## まとめ

工数見積もりという「職人芸」を、構造化ルールとClaude Code Agentで再現可能なシステムに変換しました。

ポイント：
1. **DITA的な階層構造**で機能間の関係を定義
2. **変更タイプ別の副作用リスク**で係数を調整
3. **範囲見積もり**で不確実性を正直に表現
4. **Agentとして実装**することで誰でも同じ品質で実行可能

この仕組みは、テスト工数に限らず、開発工数やレビュー工数など、様々な見積もりに応用可能です。
