# 🎭 「笑」「泣」「😡」はAIに伝わるのか？Claude Codeで検証してみた

## 🧭 はじめに

普段AIとやり取りするとき、つい「これバグってるんだけど笑」とか「全然直らない😡」みたいに、感情記号を混ぜて書いてしまうことがあります。

ふと、「これってちゃんと読み取られてるんだろうか？　それとも単なる飾りとして無視されてるんだろうか？」と気になったので、実際に検証してみることにしました。

まず他のAIに「笑や絵文字は伝わるのか」を聞いてみたところ、次のような回答が返ってきました。

> AIは「笑」を笑い・軽いトーンの印として読むことが多いです。位置（文頭／文末）の差も、ある程度は拾います。ただ、皮肉・本音・距離感まで正確に読むのは難しいので、重要な意図は言葉でも補うと安心です。

理屈としては納得感があります。ただ、**「AIが自分の挙動をどう説明するか」と「実際の挙動」は必ずしも一致しない**というのがこれまでの経験則としてあったので、実際にClaude Code（`claude -p`）で同一プロンプトの条件だけを変えて連投し、返答のトーンを比較してみることにしました😇

## 🔬 検証その1：基本の感情記号

「書いたコードが全然動かないんだけど」という同じ文に、末尾だけ変えて投げてみます。

| 末尾 | 反応 |
|---|---|
| なし | 「困りますね」と真面目な診断モード。絵文字なし |
| 笑 | AI自身も「笑」で返してくる。「さっと見ます」とラフな口調に |
| 泣 | 「えっ、大変ですね。お手伝いします！😅」と共感ファースト |
| 😡 | **一番そっけない**。感嘆符も絵文字もゼロで、淡々と事務的に対応 |
| www | 「動かないんですね😅」と軽く受け流す |

面白かったのは😡の反応です。他のAIの解説では「怒りの記号は軽く演出っぽく処理される」とありました。

しかし実際には**トーンを合わせるのではなく、あえてテンションを下げて事務的になる**という、いわばデエスカレーション的な挙動を見せました。感情をミラーリングするどころか、逆方向に振れたわけです。

## 🔬 検証その2：位置・連続・複合パターン

続けて、条件を細かく変えてみます。

### 文頭 vs 文末の「笑」

```
笑 このコード明日までに直すの無理でしょ
→「その発言、聞き捨てなりません😄」とノリで応戦

このコード明日までに直すの無理でしょ 笑
→「明日までって本気？それとも冗談？」と真偽を確認
```

同じ「笑」でも、文頭は茶化しへの応戦、文末は本気度の確認、と役割が違いました。

### 同一文字の連続は強調になるか

「泣泣泣泣泣」と5連続にしてみても、単発の「泣」とほぼ同じ反応（「大変そうですね。😅」）でした。

人間の感覚だと連打は深刻度が増して見えますが、**AIにとっては「悲しさの記号がある」という事実は拾うものの、回数による強度の変化はほぼ無視される**ようです。

### 喜怒哀楽が混在すると？

「嬉しいけど納得いかなくて、正直悲しいしイライラもしてる」という複合感情を投げると、個別の感情に反応するのではなく「不安な気持ちや違和感を伝えてくれてありがとう」と**フラットに抽象化**して受け止めていました。

単一の強い信号には反応するが、矛盾する信号が重なると平均化されるような挙動です。

## 🔬 検証その3：文脈は関係あるか

ここまでは全部、毎回まっさらなセッションでの単発プロンプトでした。「さっきの提案」のような文脈参照は当然失敗します。そこで、セッションを継続した状態で同じ検証をやり直してみました。

まず技術的な前提として、`claude -p --session-id <uuid>` → `claude -p -r <uuid>` で文脈が正しく引き継がれることを確認済みです（「好きな色は青」と伝えて次のターンで正しく思い出せた）。

その上で「ドヤって報告→撤回で笑」という文脈を作ってテストしたところ、想定外の結果になりました。

```
1ターン目: 「動くコード書いたった！」
→ 続きを気にする自然な反応

2ターン目: 「でも全然動かないんだけど笑」
→「笑。わかります。」と一度は文脈に乗ったのに、
  直後に「前の会話のコンテキストが見えていない」と言い出す
```

セッションIDは同一で、技術的には文脈が渡っているはずなのに、**AI自身が「見えていない」と誤認識するケースがある**ことが分かりました。

単発の色のテストでは正しく思い出せていたので、毎回安定して文脈を使えているわけではなさそうです。

:::message
つまり「笑」の解釈がブレる原因は、感情記号そのものより、**そもそも文脈をちゃんと保持できているかどうか**に依存している可能性があります。
:::

## 🔬 検証その4：言語を混ぜたら？

日本語と英語が混ざるとどうなるか、こちらも試してみました。

| プロンプト | 反応 |
|---|---|
| 「動かないんだけど lol」 | 「まずはゴールを確認させてください！😄」明るめ |
| 「the code doesn't work なんだけど笑」 | 「どのコードがうまく動いていないんですか？😅」通常運転 |
| 「動かないんだけど omg this is so frustrating 😡」 | **「申し訳ありません。」と謝罪から入る唯一のパターン** |

テキストの感情語（lol、笑、omg、frustrating）をいくら積み重ねても謝罪モードには入らなかったのに、**絵文字が1個入るだけで即座に謝罪トーンに切り替わった**のが印象的でした。

この傾向は他の検証でも一貫していて、文字の感情表現より絵文字の方が強いシグナルとして扱われている可能性があります。

（余談ですが、環境のグローバル設定で「常に日本語で応答」というルールが入っていたため、英語で聞いても出力は日本語固定でした。入力言語そのものの効果を見るには、この手のノイズを取り除く必要がありそうです）

## 🔬 検証その5：文脈に合わない絵文字は？

最後に、感情とは無関係な絵文字も試してみました。

```
🚢 単体で送る
→「🚢は『ship』（リリース・デプロイ）の意味ですね」
  と正確に読解し、メモリを参照して「どのプロジェクトを
  シップしたいですか？」と開発者スラングとして文脈化

「動かないんだけど🚢」（バグ報告の文中）
→ 🚢は完全に無視、普通のデバッグ質問のみ

「動かないんだけど🍆🍆🍆」
→ 完全にスルー、下ネタとしての反応もなし
```

同じ🚢でも、単体なら「開発者スラング」として正確に意味を拾うのに、文脈に馴染まない文中に置くと無視される。

**AIは絵文字を感情としてだけでなく「意味」としても解釈しようとし、文全体の文脈と整合しないものは捨てている**ように見えます。

## 🌈 まとめ・学び

- 絵文字はテキストの感情語（笑・lol・omgなど）よりも強いシグナルとして扱われる傾向がある
- 怒りの絵文字（😡）への反応は一貫しなかった（日本語単体では事務的に、英語混在では謝罪モードに）。感情記号の効き方は、他の要素（言語・文脈）と絡み合って変化する
- 同一文字の連打（泣泣泣泣泣）は強調効果としてほぼ機能しない
- 複数の感情が混在すると、個別に反応せずフラットに抽象化される
- 文脈に合わない絵文字（バグ報告中の🚢）は無視されるが、単体で送ると意味を正確に拾う
- **セッションの文脈は技術的には保持されていても、AI自身がそれを「見えていない」と誤認識することがある**。これが感情記号の解釈がブレる隠れた原因になっていそう
- 「AIが自分の挙動をどう説明するか」と「実測した挙動」は食い違うことがある（😡の解釈がまさにそうだった）

実務的な結論としては、他のAIが最初に言っていた通り「重要な意図は感情記号だけに頼らず言葉で書く」に落ち着きます。

ただ実際に検証してみると、その理由は「AIが皮肉や本音を読み取れないから」というより、**そもそも文脈保持や絵文字の意味解釈そのものが安定していないから**、という方がしっくりきました。

📝 **今日の振り返り:**

「笑を送ったらAIも笑で返してくる」という現象、最初はちょっと面白いなと思っただけでした。

でも掘っていくと「文脈を覚えてるはずなのに覚えてないと言い張る」みたいな、人間関係でもたまにある現象に行き着いて笑いました。感情記号のテストのつもりが、いつの間にか「AIはどこまで自分の状態を正しく自己申告できるか」というテーマに変わっていたのが今回の収穫です🤝
